Why is the bank progressing with this project?

Currently CBCI system is used for handling customer queries and x-selling post performing successful level checks with the customer. In various processes the customers are required to send in writer requests which customers find inconvenient to provide for any change or modification on their card account. To avoid any inconvenience to the customer & at the same time to a have more robust, authentic & non disputable customer validation process whereby limiting fraudsters intervention, an IVR authentication process has been initiated.


SYSTEM LEVEL CHANGES:

The CBCI application has been enhanced on 20th April 2012 to support this authentication project. Due to this all users would have now noticed certain additional new details displayed on the CBCI screen post login.


WHAT IS THE CHANGE NOW DUE TO AUTHENTICATION?

Currently actioning of maintenances & x-selling are done on the account post level check verification with the customer. With authentication now live, the customer validation will now be done basis One Time Password (OTP) or ATM PIN which the customer themselves will key in through their phone. Hence the customers are now assured that they account is managed in a more secured manner.

SYSTEM LEVEL MAINTENANCE:

The AOC process will allow either of the below combinations -

  1. ATM      PIN
  2. OTP      (One Time Password).
  3. Level      Checks


IVR Authentication flag can be set for either the entire call or mapped only for the particular process at link level.

  • Entire      Call – The authentication result will be      valid till the user ends the call.
  • Link      Level – The validity of authentication      result will be only for that particular link.

At PhoneBanking, we will be using both these option while at CVM team the ‘link level’ option will be used.


HOW TO PROCEED WITH AUTHENTICATION:

Customer calls PhoneBanking on a query and the process will continue as per the existing level check process. Any maintenance or fulfillment of X-sell product will require the customer to clear the new authentication process before the request can be actioned in CBCI.


Before commencing the authentication process for the customer, the Customer Services Representative will need to check whether the process requires an OTP and / or ATM PIN. (Also should check if the process allows for Level Check option to be used.)  This can be identified in the below screen.



Basis the required maintenance, the Customer Services Representative will need to check with the customer if they already have the PIN details with them handy at the moment.

                                             

For cases where the customer requires an OTP, the Customer Services Representative will need to generate an OTP for the customer by clicking on the below link.


A new pop up will now be displayed for the Customer Services Representative which will mention the last 4 digits of the mobile number to which the OTP will be despatched.


The Customer Services Representative will need to confirm this detail with the customer and then click the OK tab. The OTP will be immediately despatched only to the registered mobile number of the customer. Here the OTP will not be sent to the customer e-mail id like in VBV MS process.

AAN based AOC OTP will be different from that Credit card based AOC OTP.


  • Once      the authentication OTP is used, the OTP becomes inactive and cannot be      used once again.
  • If      the customer enters the OTP incorrectly, then the customer can use the same      OTP once again up to 3 attempts. Post the 3rd attempt the OTP      becomes invalid.


AUTHENTICATION – CALL PROCESS:

Post customer receiving the OTP, the below authentication process can be commenced by the Customer Services Representative –

  • Press      the flash button on the phone. Now the customer’s line will be on hold and      the customer will hear the IVR hold music.
  • Now      dial the short code (applicable for the respective RCC's) and the call      will get connected to the IVR. Customer is still on hold here.
  • The      IVR will prompt the team member to enter the caller id as displayed on the      CBCI screen. The user will need to enter the caller id followed by the      FLASH key.


  • On      entering the caller id and the FLASH key the customer will be conferenced      on the call and the IVR will voice out the welcome script as “Welcome to HDFC Credit Card      authentication system” 
  • There      after the customer will need to follow the IVR prompts and key in the      details as required. The Customer Services Representative will remain      connected on call and will be in the conference and will be able to hear      the customer and the IVR prompts. If required the Customer Services      Representative can also speak with the customer and guide the customer on      the details that is to be entered.

IVR AUTHENTICATION RESPONSE CODES

IF ATM PIN USED FOR AUTHENTICATION IF OTP USED FOR AUTHENTICATION
0   – Approved 00 – Success
14 – Invalid Card 01 – Failed
91 – Server error or Processing error 02 – Invalid Password
88 – PIN failure (PIN Block error) 03 – Used & Expired Password
55 – Invalid PIN 04 – Blocked Password
91 – Authentication failure (Time Out) 05 – Bad Input
62 – Blocked Card 06 – DB Error
54 – Expired Card 07 – Internal Server Error
92 – Routing Problem (Eg – BIN not present) 08 – No valid Password in system
09 – Unused & Expired
10 – Blocked link data
99 – General Error

FAQ’s:

Q1 What is the validity   period of the authentication OTP?
A1 2 hours from the   time of OTP generation
Q2 Is the authentication OTP and VBV MS OTP the   same?
A2 No, Authentication OTP is a unique OTP and   will not be the same as VBS MS OTP.
Q3 Is the authentication   OTP and IVR OTP the same?
A3 No, Authentication   OTP is a unique OTP and will not be the same as IVR OTP.
Q4 Where ever there is level check option, can we   directly choose this option and proceed with the maintenance / x-sell?
A4 Level Check option built in the system to support the customers   on their requests where ever the customer is unable to clear authentication   either through OTP or ATM PIN. Hence should not be used as a direct option.
Q5 Who can generate an authentication OTP?
A5 Authentication OTP can be generated to the   customers registered mobile number only through CBCI. There is no option for   the customer to generate  the   authentication OTP by themselves.
Q6 Should customers be advised to call back after   they generate an authentication OTP?
A6 The authentication OTP usually gets delivered   to the registered mobile number of the customer in around 5 – 10 sec. In case   of any delays is experienced by the customer in receiving the authentication   OTP, then the customer can call back within 2 hours post receiving the   authentication OTP.
Q7 In case if the customer calls back within 2 hours   for another request to be actioned on the account, then can the same password   be used?
A7 An OTP once used will get inactive, hence the   customer will require a new authentication OTP for the new requests.
Q8 What level check must be cleared by the customer   for generation of authentication OTP?
A8 Customer will need to clear level 2 for   generation of authentication OTP.
Q9 How can we identify if the authentication link   access is provided for a user or not?
A9 While accessing an account through new contact   and when clicking on one of the process link which is enabled for AOC, then   the AOC screen will be displayed instead of the regular level checks screen.   Eg. COA link
Q10 Is Level 1 & 2 checks mandatory on every   call even when we do authentication for eg.. COA?
A10 Screens which prompt for level check, will   need be cleared by the customer in order for the team member to proceed with   the required actions.
Q11 How can we identify if the authentication OTP   is valid or not?
A11 If the customer is not sure if the   authentication OTP is valid or not, then the team member can generate a OTP   for the customer. If the OTP is the same then the OTP is still valid if not   the new OTP can be used.
Q12 How many attempts will the system allow for   OTP validation in case if the customer has entered the OTP incorrectly?
A12 If the customer is not sure if the   authentication OTP is valid or not, then the team member can generate a OTP   for the customer. If the OTP is the same then the OTP is still valid if not   the new OTP can be used.
Q13 In case if the agent enters the wrong caller   ID during IVR authentication then can the agent complete the required   maintenance for the customer?
A13 Rights to the agent to do the required   maintenance will not be enabled in CBCI.
Q14 If the mobile number is changed on call, will   the OTP now be instantly sent to the new updated mobile number?
A14 No. The new mobile number will get updated in   the OTP Server only by end of day.
